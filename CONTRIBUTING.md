# 贡献指南

本仓库服务需要核验 LLM 新闻、理解工程方法和设计实践验证的读者。贡献以人工核验和知识沉淀为主，不引入自动抓取。

## 建议如何进入仓库

1. 新线索先写入 [inbox/news-suggestions.md](inbox/news-suggestions.md)，状态设为 `待核验`。
2. 疑似漏收先写入 [gaps/missed-news.md](gaps/missed-news.md)，只写候选与待核验问题，**禁止直接写结论**。
3. 核验通过后，按可信度分层（A/B/C）决定是否沉淀。

## 可信度如何标注

| 等级 | 来源 | 可否直接进专题/场景 |
| --- | --- | --- |
| A | 论文、官方文档、官方规范、项目仓库 | 可以 |
| B | 厂商工程博客、维护者文章、会议演讲 | 可以，需说明适用边界 |
| C | 社区文章、媒体、二手总结 | 不可以；只作线索，回到 A/B 核验 |

## 何时可以写 topic / scenario / lab

- **topic**：概念稳定、有代表资料、能写清边界与风险。
- **scenario**：有目标、输入输出、流程、验收；企业场景还需适用边界、资料地址、最小数据表。
- **lab**：有明确问题、本地可复现（或写清外部依赖）、输出可被 Harness 比较。

## 「已沉淀」完成定义

一条建议标为 `已沉淀` 时，至少完成：

1. 在 [sources/README.md](sources/README.md) 增加一行（含对应专题 / 对应场景）；
2. 反链到至少一个 `topics/*/README.md` 或 `scenarios/*.md`（或 `labs/`）；
3. 在 inbox 的「沉淀位置」列写明路径。

高价值单篇应另建 `sources/cards/` 证据卡，写清来源事实、能够支撑的判断、不能直接支撑的结论和工程迁移位置。

## PR 检查清单

- [ ] 未把 gaps 候选写成既定事实
- [ ] 新资料有「为什么重要」与核验日期
- [ ] 专题、来源、场景和实验之间至少建立一条可追溯关系
- [ ] 资料卡区分来源事实、仓库判断和适用边界
- [ ] 若改 Pages，只动 `gh-pages` 的 `index.html` / `assets/*`，保持零构建
- [ ] 新 lab 含 `README.md`、`data/`、`src/` 或等价说明、`reports/`
