# Prompt 评审工作流

## 目标

验证一个 Prompt 工程流程是否能把模糊需求转成可执行、可评审、可迭代的任务接口。

## 适用专题

- Prompt 工程
- Harness 工程

## 输入

- 用户输入：一段含糊需求，例如“帮我写一份竞品分析”。
- 系统上下文：目标受众、可用资料、输出限制、禁止事项。
- 评审标准：准确性、完整性、结构、证据、语气和长度。

## 输出

- 澄清后的任务说明。
- Prompt 草案。
- 输出格式约束。
- 自评审 rubric。
- 最终回答或样例输出。

## 流程

1. 抽取用户目标、受众、交付物和隐含约束。
2. 对缺失信息给出澄清问题，或在无法提问时列出假设。
3. 生成包含任务、上下文、约束、格式和失败处理的 Prompt。
4. 用 rubric 检查 Prompt 是否可执行。
5. 用同一输入跑一次样例输出，记录失败点。

## 验收标准

- Prompt 中明确任务目标、输入、输出和边界。
- 输出格式可被人工或脚本检查。
- 评审标准能指出具体失败，而不是只说“好/不好”。
- 至少记录一个失败样例和修订建议。

## 失败模式

- 只优化措辞，没有定义输出契约。
- 示例和真实任务不一致。
- 把用户输入直接拼入指令区，造成注入风险。
