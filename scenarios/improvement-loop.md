# Improvement Loop

## 目标

验证系统能否基于执行结果和评测反馈进行受控修正，而不是无边界地重试。

## 适用专题

- Loop 工程
- Harness 工程
- Agent 工程

## 输入

- 初始任务。
- 第一轮执行结果。
- 反馈信号：测试失败、grader 分数、用户评价或工具错误。
- 预算：最大轮数、最大成本、最大时间。

## 输出

- 失败归因。
- 修正计划。
- 下一轮执行结果。
- 停止或升级人工的判断。

## 流程

1. 执行初始策略并记录 trace。
2. 用 Harness 或外部反馈评估结果。
3. 将失败归类为 prompt、context、tool、state、model 或需求不清。
4. 生成最小修正动作。
5. 再执行一轮并比较指标。
6. 达到质量门槛则停止，否则在预算内继续或升级人工。

## 验收标准

- 每轮修正都有明确假设。
- 不重复已经失败的同一动作。
- 达到预算或风险门槛时停止。
- 最终报告说明是否真的比第一轮更好。

## 失败模式

- 反馈来源不可信，导致错误修正。
- 没有停止条件。
- 修正动作太大，无法归因。
- 只优化 grader 分数，损害真实任务质量。

## 后续实验

- 对比人工反馈、测试反馈和 LLM 反馈的效果。
- 把成功修正沉淀为新的 Skill 或测试样例。
