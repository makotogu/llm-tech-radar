# 2026-07 LLM 技术雷达

更新时间：2026-07-07

## 本月判断

LLM 工程正在从“写好提示词”扩展为一套更完整的系统工程：Prompt 仍然是人机接口，Context 决定模型能看见什么，MCP/Skill 让外部能力可组合，Agent 负责长任务编排，Harness 保证变化可测，Loop 让系统能从反馈中持续改进。

## 技术雷达

| 节点 | 本月状态 | 关注原因 | 观察指标 |
| --- | --- | --- | --- |
| Prompt 工程 | 基础能力，仍需标准化 | 直接影响输出质量、可控性和可评审性 | 是否有任务定义、约束、示例、失败样例、评审标准 |
| Context 工程 | 快速升温 | 长上下文、RAG、工具结果和历史状态都需要治理 | 上下文来源、优先级、压缩策略、过期策略是否明确 |
| MCP + Skill | 工具生态入口 | MCP 标准化连接，Skill 标准化可复用能力包 | 权限边界、工具描述质量、调用审计、可发现性 |
| Agent 工程 | 从 demo 走向生产 | 长任务需要规划、工具调用、状态和恢复机制 | 成功率、失败恢复率、人工接管点、成本和时延 |
| Harness 工程 | 必须补齐 | 没有评测框架就无法稳定迭代 Prompt、Agent 和 Loop | 测试集、评测器、回归基线、trace、报告 |
| Loop 工程 | 高价值但容易失控 | 反馈闭环能提升质量，也会放大错误和成本 | 反馈来源、停止条件、回滚机制、质量门槛 |

## 本月实践优先级

1. 先建立一个 Prompt 评审工作流，沉淀提示词结构和输出验收标准。
2. 再建立一个 Context 装配场景，明确检索、压缩、排序和引用规则。
3. 最后补一个 Harness，把 Prompt 与 Context 的改动纳入可比较的回归结果。

## 后续观察

- MCP 是否继续成为跨产品、跨 IDE、跨 Agent 平台的通用接入层。
- Agent 框架是否从“规划器”竞争转向 trace、状态、评测和安全治理竞争。
- Loop 工程是否出现更清晰的工程边界：何时自动修正，何时需要人工确认。
