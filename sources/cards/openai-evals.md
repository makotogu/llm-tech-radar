## 一句话价值

这份文档把评测拆成可版本化的测试标准、数据、运行和结果，适合用来检查内部 Harness 是否具备最小闭环。

## 文档关注点

- 评测需要先定义任务成功标准，再准备代表性测试数据和 grader。
- 一次运行应关联明确的候选配置和结果，才能用于回归或候选比较。
- grader 本身也需要人工样例校准，不能把自动分数视为天然真值。

## 这份资料能支撑什么

- 可以支撑：数据集、候选、grader 和运行结果需要独立版本化。
- 可以支撑：每次模型、Prompt 或工具变更都应触发可比较回归。
- 不能直接支撑：特定 API 是建立 Harness 的唯一方式。
- 不能直接支撑：单个汇总分数足以决定高风险系统上线。

## 工程迁移

- 逐样例保存输入、期望、候选输出、grader 结果、失败标签和人工复核。
- 汇总报告同时显示平均分、高风险失败和成本时延。

## 注意事项

- 官方 API 和产品导航会持续变化，仓库保留的是评测对象和证据结构。
- 自动 grader 的稳定性、偏差和与人工一致率必须单独评测。
