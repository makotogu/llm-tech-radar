## 一句话价值

这份文档把 MCP 从“工具协议”拉回真实信任边界，要求实现者同时考虑授权、凭证、会话、代理和本地服务风险。

## 文档关注点

- MCP 安全不能只检查 tool schema，还要检查客户端、服务端、授权服务器和下游 API 之间的数据与凭证流。
- 安全控制需要遵守最小权限、明确授权边界和凭证隔离，避免把上游 token 无条件传递给下游服务。
- 本地 MCP server 仍然是可执行软件，需要安装来源、更新机制和运行权限控制。

## 这份资料能支撑什么

- 可以支撑：MCP 接入前必须画出信任边界和凭证流。
- 可以支撑：工具参数校验不能替代身份、授权和动作审批。
- 不能直接支撑：使用 MCP 协议本身就能获得安全隔离。
- 不能直接支撑：所有实现都应使用同一种认证或部署方式。

## 工程迁移

- 为每个工具登记数据范围、权限、凭证来源、审批条件和审计字段。
- 只读、建议型写入和直接写入使用不同门禁。
