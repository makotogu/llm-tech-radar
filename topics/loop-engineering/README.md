# Loop 工程

## 定义

Loop 工程是设计“执行、观察、反馈、修正、再执行”的闭环系统。它关注的不只是单次回答质量，而是系统如何利用 trace、评测结果、用户反馈和环境反馈持续改进。

## 要解决的问题

- 一次生成无法满足复杂任务，需要迭代修正。
- Agent 执行中会遇到错误，需要从反馈中恢复。
- 改进过程如果没有停止条件和质量门槛，会造成成本失控或错误放大。

## 关键概念

| 概念 | 解释 | 需要避免的误解 |
| --- | --- | --- |
| Feedback signal | 来自测试、工具、用户、环境或模型评审的反馈 | 所有反馈都同等可靠 |
| Repair action | 基于反馈进行重试、改 prompt、改上下文或改工具调用 | 盲目重试 |
| Memory | 保存对后续尝试有价值的经验 | 把所有历史都塞回上下文 |
| Stop condition | 何时停止自动循环 | 无限循环直到看起来成功 |
| Quality gate | 进入下一阶段前的最低质量门槛 | 没有回滚路径 |

## 内容摘取

- Loop 把单次生成变成“执行 -> 观察 -> 反馈 -> 修正 -> 再执行”的系统过程。
- 反馈信号可以来自测试、用户、工具、环境、评测器或模型自评，但不同反馈可靠性不同。
- Repair action 不只是重试，还包括改 prompt、换上下文、调整工具调用、补样例、降级或升级人工。
- Stop condition 是生产 Loop 的核心：没有次数、预算、质量门槛和回滚条件，循环会失控。
- Memory 应该保存可复用经验和失败模式，而不是把所有历史原样塞回上下文。
- Loop 的产物应回流到 Skill、测试集、资料库或 runbook，避免同类错误反复发生。

## 场景与用途

| 场景 | 用途说明 | 重点产物 |
| --- | --- | --- |
| 代码测试失败修复 | 根据测试失败、lint、类型错误自动定位并修正代码 | 失败分类、补丁、重测记录 |
| RAG 答案改进 | 根据低分样例调整检索、排序、压缩或引用策略 | 错误归因、上下文变更、对比结果 |
| Agent 调用恢复 | 工具失败、权限失败或参数错误后自动诊断和重试 | 调用 trace、恢复策略、升级条件 |
| Prompt 自动优化 | 用 Harness 失败样例生成候选 prompt，再评测筛选 | 候选版本、分数变化、回滚点 |
| 用户反馈闭环 | 把用户纠错转成新样例、规则或知识库更新 | 反馈标签、修复记录、验证样例 |
| 文档持续改进 | 根据阅读反馈和遗漏清单补充资料、专题和场景 | 缺口清单、更新 diff、验收说明 |
| 生产质量门禁 | 失败超过阈值时自动停止发布或要求人工确认 | 停止条件、告警、回滚动作 |
| Grader 校准循环 | 将人工复核差异反馈给评分规则和模型裁判提示 | 偏差清单、校准版本、一致率变化 |
| 成本优化循环 | 根据 trace 找出高成本调用，调整缓存、压缩或模型路由 | 成本归因、优化动作、质量回归结果 |
| 安全策略修复循环 | 将红队绕过样例转成规则、prompt 或工具权限修复 | 绕过样例、修复记录、复测结果 |
| 数据标注改进循环 | 根据低置信样例和评审分歧补充标注指南与样例 | 分歧样例、指南变更、复核一致率 |

## 边界与风险

- 自动循环必须有预算、次数和质量门槛。
- 反馈错误会让系统越改越差。
- 高风险动作应在 Loop 中设置人工确认点。

## 最小实践场景

- 实验目标：用同一批失败样例验证自动修正是否真的优于单次执行。
- 实验材料：准备 10 个任务、初始执行结果、失败反馈、2 条修正策略和固定停止条件。
- 实验步骤：运行初始策略；记录 trace 和评分；按失败类型选择修正动作；最多重试 2 轮；比较前后结果。
- 记录方式：保存每轮输入、反馈、修正理由、重试次数、分数变化、成本和是否触发人工升级。
- 验收标准：失败能被分类，重试次数受控，最终结果优于单次执行；无收益或高风险样例能及时停止。
