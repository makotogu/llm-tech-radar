# Agent 工程

## 定义

Agent 工程是构建能围绕目标持续执行任务的 LLM 系统：它需要规划、调用工具、读取和更新状态、处理失败、暴露 trace，并在必要时请求人工接管。

## 要解决的问题

- 长任务无法一次完成，需要拆解、执行、观察和修正。
- 工具调用结果不稳定，需要错误处理和恢复。
- 多步骤系统如果没有 trace 和状态，很难调试、评测和复现。

## 关键概念

| 概念 | 解释 | 需要避免的误解 |
| --- | --- | --- |
| Plan | 对任务步骤和依赖的显式表达 | 计划越长越好 |
| Tool use | 调用外部能力获取信息或执行动作 | 所有任务都应该工具化 |
| State | 任务进度、历史决策、环境反馈和中间产物 | 只保存自然语言聊天记录 |
| Guardrail | 输入、输出和动作的安全边界 | 事后人工 review 替代运行时控制 |
| Trace | 可观察的决策、调用和结果链路 | 只记录最终答案 |

## 演进时间线

| 时间 | 事件 | 判断 |
| --- | --- | --- |
| 2022 | ReAct 提出 reasoning 与 acting 交错 | Agent 工作流有了清晰研究范式 |
| 2023-2024 | AutoGPT、LangChain、工具调用生态扩散 | Agent demo 很多，但生产稳定性不足 |
| 2025-2026 | Agents SDK、评测、trace、guardrails 进入主线 | Agent 工程重点转向可靠性和可观察性 |

## 适用场景

- 代码修改、测试、修复和提交。
- 多系统查询和业务流程自动化。
- 需要计划、执行、检查和人工确认的办公任务。

## 边界与风险

- Agent 不适合无约束地自动执行高风险操作。
- 规划和执行都可能错，需要 Harness 和 Loop 兜底。
- 工具权限过大时，单次误调用会变成系统事故。

## 最小实践场景

- 输入：一个跨文件、跨工具的任务。
- 输出：计划、执行日志、工具调用记录、最终产物和失败恢复说明。
- 流程：理解目标、生成计划、执行步骤、观察结果、修正、完成。
- 验收标准：可复现、可中断、可观察、失败时能给出下一步。
