# Agent 工程

## 定义

Agent 工程是构建能围绕目标持续执行任务的 LLM 系统：它需要规划、调用工具、读取和更新状态、处理失败、暴露 trace，并在必要时请求人工接管。

## 要解决的问题

- 长任务无法一次完成，需要拆解、执行、观察和修正。
- 工具调用结果不稳定，需要错误处理和恢复。
- 多步骤系统如果没有 trace 和状态，很难调试、评测和复现。

## 关键概念

| 概念 | 解释 | 需要避免的误解 |
| --- | --- | --- |
| Plan | 对任务步骤和依赖的显式表达 | 计划越长越好 |
| Tool use | 调用外部能力获取信息或执行动作 | 所有任务都应该工具化 |
| State | 任务进度、历史决策、环境反馈和中间产物 | 只保存自然语言聊天记录 |
| Guardrail | 输入、输出和动作的安全边界 | 事后人工 review 替代运行时控制 |
| Trace | 可观察的决策、调用和结果链路 | 只记录最终答案 |

## 内容摘取

- Agent 的核心不是“模型自动做事”，而是围绕目标组织计划、工具、状态、观察、恢复和人工接管。
- ReAct 类方法强调推理和行动交错：模型先判断下一步，再调用工具观察环境，再更新计划。
- 生产 Agent 需要明确任务边界、权限边界、失败恢复路径和可观察 trace。
- State 不能只保存聊天记录，还要保存任务进度、已尝试动作、产物位置、阻塞原因和下一步。
- Guardrail 应该覆盖输入、输出和动作；高风险操作要在执行前确认。
- Agent 的可靠性依赖 Harness 和 Loop：没有评测与反馈，自动执行会快速积累错误。

## 场景与用途

| 场景 | 用途说明 | 重点产物 |
| --- | --- | --- |
| 代码任务执行 | 修改代码、运行测试、根据失败继续修复并提交 | 计划、diff、测试记录、失败恢复 |
| 调研与资料整理 | 搜索资料、筛选来源、归纳观点、生成报告 | 来源列表、摘要、可信度判断 |
| 业务流程自动化 | 跨 CRM、工单、邮件、表格执行多步流程 | 工具调用记录、人工确认点 |
| 数据分析助理 | 读取数据、生成查询、解释结果、输出结论 | notebook、查询日志、图表和结论 |
| 运维诊断 | 读取日志、查询监控、定位异常、提出修复步骤 | trace、假设列表、处置建议 |
| 文档维护 | 根据变更更新 README、规格、Runbook 和模板 | 变更摘要、链接检查、审阅清单 |
| 个人助理任务 | 安排日程、整理待办、准备会议材料 | 状态记录、权限审批、最终清单 |
| 数据迁移助理 | 规划迁移步骤、生成校验脚本、跟踪异常和回滚条件 | 迁移计划、校验报告、风险清单 |
| 依赖升级 Agent | 读取 changelog、修改配置、修复兼容问题并运行回归测试 | 升级 diff、破坏性变更说明、测试记录 |
| 采购比价 Agent | 收集供应商资料、规格、报价和风险，生成对比建议 | 来源表、评分矩阵、待确认问题 |
| 合规检查 Agent | 对文档、流程或代码变更执行规则检查并提出整改项 | 检查 trace、违规证据、整改建议 |

## 边界与风险

- Agent 不适合无约束地自动执行高风险操作。
- 规划和执行都可能错，需要 Harness 和 Loop 兜底。
- 工具权限过大时，单次误调用会变成系统事故。

## 最小实践场景

- 实验目标：让 Agent 完成一个跨文件任务，并验证它的计划、状态、工具调用和失败恢复是否可观察。
- 实验材料：准备一个小型代码仓库、3 个变更需求、1 个会失败的测试和 1 个需要人工确认的操作。
- 实验步骤：生成计划；逐步执行；每步记录观察结果；遇到失败时分类原因；修正后重跑验证。
- 记录方式：保存计划、工具调用、状态变化、失败日志、修复 diff、测试结果和人工确认记录。
- 验收标准：任务可复现、可中断、可观察；失败时能给出下一步；最终产物有验证证据。
