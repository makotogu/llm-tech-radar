# Labs

这里放后续可运行实验、demo、评测脚本和结果报告。首版先保留结构，不放未验证代码。

## 推荐实验顺序

1. `prompt-review-workflow/`：提示词需求澄清、输出约束和评审。
2. `context-assembly-rag/`：多来源上下文装配、压缩和引用。
3. `agent-tool-runner/`：带工具调用、状态记录和失败恢复的任务型 Agent。
4. `eval-harness/`：对 Prompt、Context 和 Agent 策略做回归评测。
5. `improvement-loop/`：用 trace 与 eval 结果驱动下一轮修正。

## 大节点实验说明

| 大节点 | 简单实验 | 输入材料 | 主要输出 | 验收方式 |
| --- | --- | --- | --- | --- |
| Prompt 工程 | 把模糊需求改写成结构化 prompt，并对比改写前后输出 | 5 条用户需求、输出格式要求、评审标准 | 修订 prompt、输出样例、失败分析 | 至少 4/5 个样例在格式和约束满足度上更稳定 |
| Context 工程 | 构建小型 RAG 上下文装配流程，比较直接塞材料和检索排序压缩 | 10 篇短文档、5 个问题、模拟工具输出 | 上下文清单、引用回答、丢弃原因 | 关键事实可追溯来源，无关材料不进入最终上下文 |
| MCP + Skill | 把重复任务拆成工具和 Skill 说明，验证模型能否正确选择能力 | 2 个只读工具、1 个写操作工具、1 份资源文档 | 工具 schema、Skill 触发规则、调用记录 | 敏感操作需要确认，调用参数可审计 |
| Agent 工程 | 让 Agent 完成一个跨文件任务并记录计划、状态和失败恢复 | 小型代码仓库、3 个需求、1 个失败测试 | 执行计划、工具日志、修复 diff、测试结果 | 任务可复现、可中断、失败时能给出下一步 |
| Harness 工程 | 比较两个候选策略，输出分数、失败分类和推荐结论 | 20 条样例、2 个候选策略、grader、人工复核表 | 评分报告、trace、失败标签、推荐理由 | 能重复运行，能解释策略选择，关键失败不被平均分掩盖 |
| Loop 工程 | 用失败反馈驱动最多两轮自动修正，比较前后结果 | 10 个任务、初始结果、失败反馈、停止条件 | 修正计划、每轮结果、成本和停止判断 | 重试次数受控，结果优于单次执行，高风险样例能停止 |

## 每个实验建议包含

- `README.md`：实验目标、运行方式、输入输出、验收标准。
- `data/`：小型样例数据或链接说明。
- `src/`：最小可运行实现。
- `reports/`：实验记录、指标、截图或 trace。

## 进入 Labs 的条件

- 有明确问题和验收标准。
- 能在本地复现，或者清楚说明依赖的外部服务。
- 输出能被 Harness 复用或比较。
